# 场景一（三阶段）实施计划

目标：把“售前自动答疑 + 人工兜底”从当前 MVP 骨架，推进到可小规模商用。

## 阶段一：闭环可用（P0）

### 阶段目标
- 自动答疑不再是固定文案，具备最小真实问答能力
- 登录与会话安全达到可用标准
- 形成“自动答疑 -> 转人工 -> 回流候选”的稳定闭环

### 具体工作
- 鉴权真实化
  - 登录从 `DEMO_USER` 切换到 `users` 表校验
  - 接入密码哈希校验（bcrypt/argon2）
- Token 续期
  - 增加 refresh token 签发与刷新接口
  - 前端接入续期逻辑（到期前刷新）
- 自动答疑最小链路
  - 接入知识检索 top-k
  - 拼装上下文并调用模型生成答案
  - 返回引用和置信度字段
  - 低置信度自动触发 handoff

### 验收标准
- `login -> me -> knowledge-sources -> chat -> handoff` 全链路可跑通
- 自动回答非固定文案，能返回引用片段
- token 过期后可续期，不强制频繁重新登录

---

## 阶段二：可调优、可运营（P1）

### 阶段目标
- 让运营和技术都能定位“答得好/答得差”的原因
- 知识处理任务从手工触发升级为后台异步
- 支持至少一个真实外部渠道接入

### 具体工作
- 检索调试能力
  - 实现 `/knowledge-chunks` 与 `/retrieval/debug`
  - 输出召回明细、排序分数、最终上下文
- Worker 异步任务闭环
  - 跑通“导入 -> 切块 -> 索引/向量化 -> 状态回写”
  - 失败重试和任务状态可见
- Connector 单渠道接入
  - 接通 1 个 webhook 渠道（建议飞书/企微）
  - 完成签名校验、消息入站、会话创建
- 控制台产品化增强
  - 优化空态提示、错误提示、关键按钮引导

### 验收标准
- 运维/运营可用调试接口快速定位召回问题
- 新知识源可自动进入异步处理，不依赖人工手动链路
- 外部渠道消息可进入会话并触发答疑/转人工

---

## 阶段三：可商用与可扩展（P1-P2）

### 阶段目标
- 具备最小商业化能力（配额管控）
- 具备上线后的安全与稳定性底座
- 为后续规模增长准备数据治理能力

### 具体工作
- 最小配额与用量
  - 实现 `quota/check` 拦截逻辑
  - 提供简化 `plan/usage` 展示
- 安全加固
  - 默认密码替换流程与启动校验
  - 内部路径白名单与最小权限账号
  - 关键动作审计日志补全
- 数据治理与运维
  - 迁移脚本版本化（可回滚）
  - 备份恢复流程
  - 慢查询/5xx 告警
  - 聊天与日志数据归档策略（热冷分层）

### 验收标准
- 可按租户执行最小配额控制
- 出现异常有告警、有追踪、有恢复手段
- 数据规模增长下，查询与存储成本仍可控

---

## 里程碑建议
- 阶段一：1-2 周
- 阶段二：1-2 周
- 阶段三：1 周

总计：约 3-5 周（单人开发节奏）。
